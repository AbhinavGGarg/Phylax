Architecture & security

How the boundary holds.

Phylax splits the world into partner spaces and one neutral control plane, separated by a trust membrane. The privacy-preserving computation happens on the partner side, in SecretFlow. InsForge is the neutral control plane that makes that computation a usable, auditable product. Only opaque commitments, permitted aggregate findings, and hashes ever cross into the middle.

InsForge feature map

The neutral control plane, feature by feature.

InsForge is not a database Phylax happens to use — it is the trust boundary, the auditable ledger, the live incident stream, and the deploy target. Every capability maps to a concrete containment need.

Auth + RLS
6 roles — platform_admin · partner_admin · analyst · operator · approver · auditor. Authenticated members get scoped, read-only access; every write flows through an edge function under the admin key, authorized from membership (never a client-supplied org id). Tenant isolation is asserted by test: a FinTrust user cannot read SwiftCart's private signal batches or commitments.
Postgres
13 tables behind a monotonic, server-enforced state machine — draft → awaiting_parties → running → protected_match → risk_scored → awaiting_approval → actioned → receipted. An illegal transition is rejected for every role, including admin. The audit ledger and the finding tables are append-only by trigger, and raw_records_shared = 0 is a hard DB invariant.
Edge Functions
8 functions — submit-signal-manifestcreate-detection-rundispatch-party-runreceive-worker-callbackrequest-actionapprove-actionissue-receiptverify-receipt. Worker callbacks are authenticated with HMAC-SHA256 over the exact signed envelope, a timestamp freshness window, and a single-use nonce ledger for hard replay protection.
Realtime
Live progress on run:{id} and org:{id} channels, carrying safe metadata only — phase, derived score, cardinality, and the always-zero raw counter, never a signal or identifier. Channel-subscription RLS ensures only authorized viewers can subscribe.
Storage
A private phylax-artifacts bucket holds input packages, model artifacts, signed run reports, and receipts. The database keeps only bucket/key pointers plus checksums — never the raw evidence itself.
Custom Compute
Dockerized SecretFlow party services (Linux · Python 3.10) run the long-lived protected-compute agents beside each org's data, executing that party's share of the PSI and secure aggregation.
Backend Branching
A schema-only security branch validates RLS and state-machine changes in isolation before they are applied to the production control plane.
Sites
The operator console and this site deploy on InsForge Sites, served from the same project that governs the data — one control plane, front to back.
SecretFlow feature map

The privacy-preserving computation, on the party side.

SecretFlow performs the cryptography; it never touches the neutral plane. What leaves a party is only what the protocol is designed to output.

ECDH PSI
Private set intersection across three parties — ECDH_PSI_3PC (and ECDH_PSI_2PC for two), pinned to secretflow 1.10.0b1 / SPU 0.9.x. The coordinator relays fully-blinded group elements and learns only the intersection; non-matching tokens are never revealed.
Secure aggregation
Additive secret sharing over a Mersenne-prime field computes consortium sums, means, and moments for the permitted aggregate features. No party — and not the coordinator — ever sees another party's individual value; only the aggregate leaves the protocol.
Federated risk model
A real numeric model — phylax-risk 1.0.0, logistic regression over five features (distinct_parties, matched_signatures, total_events, mean_velocity_z, temporal_alignment) — scores the matched cluster from those aggregates alone.
Local backend
A pure-stdlib 2048-bit MODP DDH-PSI (RFC 3526) implements the same cryptographic family — commutative encryption in a DDH-hard group — for reproducibility and to run anywhere. It yields identical results to the SecretFlow SPU backend.
Trust boundary · demo vs production

Exactly where the demonstration ends.

The protocol and the product are real and running. The infrastructure isolation is modeled, not proven — and saying so plainly is part of the security posture.

Demo · todayfunctional demonstration
  • Synthetic data generated from a fixed seed — no real customers, messages, or accounts.
  • All three parties plus control run in a single InsForge account (one topology).
  • Proves the protocol, the state machine, and the product end to end.
  • Not proof of adversarial infrastructure isolation — the parties are not mutually distrusting hosts here.
Production · designindependent parties
  • Each org runs its own InsForge project, its own party service, and independent key material.
  • The OPRF key is jointly governed / threshold — no single party can tokenize alone.
  • Per-org Ed25519 signing; private keys never enter the control plane.
  • The neutral plane still centralizes zero raw records — the invariant does not change.
Not described as "encrypted end-to-end" — it is PSI + MPC, not E2EE No legal or compliance certification is claimed No automatic accusations — every action needs a human at the target org
What is real

The checklist, without the hand-waving.

Running now

  • 2048-bit MODP DDH-PSI executes locally; SecretFlow ECDH_PSI_3PC in production
  • Trained logistic-regression risk model — not an LLM, not a random score
  • Server-enforced monotonic state machine + append-only audit ledger
  • RLS tenant isolation across partner-private tables
  • raw_records_shared = 0 as a hard DB invariant; HMAC + nonce replay guard on callbacks

Honestly still staged

  • Single-account demo topology — not isolated, mutually-distrusting infrastructure
  • Synthetic data only — no real customer signals
  • No third-party security audit or compliance certification
  • Threshold OPRF-key governance is designed, not yet deployed

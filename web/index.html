The problem

Scam rings are cross-platform. Every platform sees a splinter.

A single ring launders through a bank, recruits on a marketplace, and sprays links over a messenger. No one platform sees enough to act — and the raw evidence that would connect the dots is exactly the data they are not allowed to pool.

How Phylax works

A five-step protected pipeline. Data stays home; only proof moves.

Every step is engineered so that what crosses the trust membrane is opaque or aggregate — never a raw record. The numbering is earned: each stage strictly follows the last.

01

Local signals

Each party derives anomaly signals over its own data, inside its own perimeter. A bank's payout velocity, a marketplace's off-platform steering, a messenger's link bursts — computed locally, under that org's own row-level security.

on-prem crosses membrane: nothing
02

OPRF-tokenized commitments

Candidate campaign signatures — wallets, domains, handles — are turned into opaque commitments under a jointly-governed key. No party can enumerate another's inputs; the control plane stores commitments, never values.

oprf-ristretto255 + keyed-normalize crosses membrane: opaque commitments
03

Protected match

An ECDH / DDH private-set-intersection runs across the three party agents. The coordinator relays fully-blinded group elements and learns only the intersection — the shared signatures. Non-matching tokens are never revealed to anyone.

ECDH_PSI_3PC · 2048-bit MODP DDH crosses membrane: opaque group elements
04

Federated risk score

A real trained logistic regression scores the cluster over five permitted aggregate features — produced by secure aggregation, never by reading raw records. Not an LLM, not a random number: a numeric model with a published feature spec and a params hash.

phylax-risk 1.0.0 crosses membrane: 5 aggregate features
05

Human-approved action · signed receipt

Phylax recommends the smallest proportionate response — hold a payout, quarantine a listing, warn recipients. A human at the target org approves it; nothing executes automatically. The decision is sealed into an append-only ledger and a signed receipt.

Ed25519 receipt crosses membrane: signed decision hash
Real, not theater

What is actually running — and what is honestly still a demo.

Security products earn trust by being precise about their own limits. Here is the honest boundary between the cryptography that runs and the topology that is staged for the hackathon.

The protocol execution is real

Protected match runs a genuine ECDH / DDH private-set-intersection: RFC 3526 2048-bit MODP locally, and a 1024-bit MODP group inside the InsForge edge function for the hosted demo. Same cryptographic family, identical results.

The risk model is a trained model

A logistic regression over five aggregate features, fit offline by gradient descent and verified to reproduce from seed. It is not an LLM and not a random score — it ships with a feature spec and a params hash.

The control plane stores zero raw records

The neutral InsForge project holds commitments, aggregate findings, hashes, and receipts — never a message, identity, account number, watchlist, or vector. raw_records_shared = 0 is a hard database invariant, enforced by trigger.

The topology is staged

The demo uses synthetic data and runs the three parties plus control in a single account. That is a functional demonstration of the protocol and the product — not proof of adversarial infrastructure isolation. Production splits each org into its own project and key material.
